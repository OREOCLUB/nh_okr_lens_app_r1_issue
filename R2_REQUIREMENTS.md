## 2. 화면 1 · 평가자 대시보드 `/r2`

**목적**: 팀원 OKR 검토 현황을 한눈에 파악하고, 검토가 필요한 팀원으로 바로 진입한다.

### 2.1 현황 카드 6종 (D4)
- 전체 팀원 / 수립완료(승인) / 작성 중 / 결재요청(제출) / 반려 / 조정요청
- 각 카드에 건수 표시, 건수 > 0 카드는 강조 처리
- **카드 클릭 → 하단 팀원 테이블을 해당 상태로 필터, 재클릭 시 해제**

### 2.2 팀원 OKR 현황 테이블
- 컬럼: 직급 · 이름 · 직군 · 제출일 · 상태 · AI 위험도 · 코칭등록 · 액션
- 기본 정렬: 상태 우선순위 **결재요청 > 반려 > 조정 > 작성중 > 완료**
- 컬럼 헤더 클릭 정렬 + 페이징 적용
- **행(또는 액션 버튼) 클릭 → `/r2/review?member={id}` 이동 (D1)** — `/r2/member`로의 링크는 만들지 않음(D5)

### 2.3 데이터·기타
- `getMembers()` Supabase 조회(실패 시 더미 폴백) + 로딩/실패 상태 표시
- 헤더 평가자 정보(이름·소속)는 세션 사용자(`okrlens_current_user`) 기준
- "검토 화면으로 이동" 버튼 → `/r2/review` 실제 라우팅 (alert 제거)

### 2.4 현재 코드와의 갭 (2026-07-03)
- 카드 4종(`r2Stats`)뿐 → 반려/조정요청 추가 필요, 클릭 필터 없음
- 테이블 컬럼 4개뿐, 정렬·페이징·행 클릭 없음
- 헤더 "박정훈 팀장" 하드코딩

## 3. 화면 2 · OKR 검토 `/r2/review` — MVP 핵심

**목적**: 팀원이 제출한 OKR을 criteria 기준으로 점검하고 승인/반려/조정요청으로 처리한다.

### 3.1 진입
- `?member={id}` 쿼리로 진입 시 해당 팀원 자동 선택 (D1)
- 직접 진입 시 검토 대상(결재요청·반려·조정) 중 첫 번째 자동 선택

### 3.2 좌측 — 검토 대상 리스트
1. 상태가 결재요청/반려/조정인 팀원 목록, 결재요청 우선 정렬
2. 필터: 상태 / 직급 / 직군
3. **팀원 클릭 → 중앙 패널이 해당 팀원의 실제 OKR로 갱신** (현재 "강동우" 하드코딩 제거 — 최대 갭)
4. 집중코칭 대상 🎯 표시 (토글 없이 표시만 — lite)

### 3.3 중앙 — OKR 상세 + AI Validation
1. **OKR 아코디언**: Objective → KR(복수) → 측정방법 → 난이도 → 실행계획. `okrs` 테이블 조회
2. **AI Validation (목업, D2)**
   - 체크 항목은 `criteria` 테이블에서 로드 — 항목 수 N 가변 (현재 9항목 하드코딩 제거)
   - `[🤖 AI 검토 실행]` 버튼 → 목업 판정 결과 표시: 항목별 통과/주의/코칭 후보 + 사유 + 종합 코멘트
   - **실제 API 호출 없음.** 판정 생성 함수를 단일 지점으로 분리해 추후 Gemini API로 교체 가능하게 설계
   - 평가자가 항목별 라디오로 판정 수정 가능, 수정 시 ✏️ 표시
   - 위험도 실시간 재계산: 코칭 후보 ≥4 상 / ≥2 중 / 그 외 하 (기준값은 criteria와 정합 유지)
3. **작년 OKR 팝업**: `okr_history` 조회 (D6). 연도 탭 + Objective/KR/난이도/최종등급/달성률/결과 + 💡 참고 자동 코멘트(난이도 상향·미달성·초과달성)
4. **반려 이력 타임라인**: 재상신 건에 1차 제출 → 1차 반려(사유) → 2차 제출 순차 표시. `okr_history` 기반 (D6)

### 3.4 우측 — 검토 처리 (통합형, D9)
1. 처리 방향 3택: 승인 / 조정요청 / 반려
2. 조정요청·반려 시 메시지 **필수 입력** validation (고정 defaultValue 제거). `✨ AI 초안` 버튼은 목업 문구 삽입(코칭 톤)
3. **처리 버튼 클릭 흐름**:
   - 승인: 즉시 저장
   - **반려·조정요청: Confirm 팝업 경유 (D7)** → 확인 시 저장
   - 저장 내용: 그 시점의 체크 결과+위험도 → `risk_analysis`, 처리 결정+메시지 → `approval_status` **동시 저장 (D9)**
   - 성공/실패 처리 + 로딩 상태 표시
   - 처리 성공 후: 좌측 리스트 상태 갱신 → 다음 검토 대상으로 자동 이동
4. `[임시 저장]` 버튼 = `risk_analysis`만 저장 (검토 도중 이탈 대비, D9)

### 3.5 현재 코드와의 갭 (2026-07-03)
- 중앙 패널 전체 하드코딩(강동우 1건), 선택 연동 없음
- 체크 9항목 하드코딩(`INITIAL_CHECKS`), "AI 재검토"는 edited 플래그 리셋만
- 작년 OKR 팝업·반려 이력 타임라인·좌측 필터 없음
- 처리·임시저장 버튼 alert만, DB 쓰기 미연동 (쓰기 RLS 정책 추가 필요 — PROGRESS 참조)
- 라디오 수정·위험도 재계산은 구현되어 있음 ✅

## 4. 화면 3 · 코칭 캘린더 `/r2/calendar` — lite + 연동(D8)

**목적**: 팀 일정(1on1·분기미팅·등록 시작/마감 등)을 월간 뷰로 확인한다.

1. 월간 캘린더 + 일정 유형 칩(1on1 코칭·분기미팅·등록시작/마감·최종평가·기타) **표시**
2. **월 이동(◀ 이전 달 / 다음 달 ▶) 구현 (D8)** — 현재 2026-07 고정 제거
3. **일정 데이터는 `eval_phases` 테이블에서 조회 (D8)** — 하드코딩 EVENTS/MONTH_LIST 대체. 조회 실패 시 더미 폴백 + 로딩/실패 상태
4. 우측 "이번 달 일정" 리스트·"1on1 미등록 팀원" 카드는 현행 유지
5. 일정 등록·1on1 등록·날짜 클릭 등록은 **P2** — "준비 중" 코칭 톤 안내 유지 허용

## 6. 구현 순서 (커밋 단위 4단계)

> 정식 이슈 분리 없이, 아래 4단계를 커밋 경계로 삼아 한 세션에서 연속 진행한다.
> **단계마다 동작 확인 + 커밋 + PROGRESS.md 기록** 후 다음 단계로 넘어간다.

### 1단계 — 쓰기 기반 마련 (전제 조건)
- `supabase/schema.sql`(또는 마이그레이션)에 쓰기 RLS 정책 추가: `okrs.approval_status` · `okrs.risk_analysis` 업데이트 허용
- `src/lib/dataAccess.ts`에 쓰기 함수 추가 (처리 저장 · 임시 저장)
- **검증**: REST/SQL로 업데이트 1건 왕복 확인
- 실패 시 뒤 단계 전부 블록되므로 반드시 먼저 완료

### 2단계 — OKR 검토 화면 (§3, 최대 물량·핵심)
- 순서: 3.2 좌측 리스트 선택 → 중앙 실데이터 바인딩(하드코딩 제거) → 3.3 criteria 로드·목업 AI Validation·작년 OKR 팝업·반려 이력 타임라인 → 3.4 처리 저장(통합형 D9 + Confirm D7)
- **검증**: 승인/반려 처리 후 **R1 홈·R3 인사이트에 상태 반영되는 연쇄까지 확인** (성공 기준: R1 제출 → R2 승인 → R3 반영)
- `?member={id}` 수신부가 이 단계에서 생겨야 3단계 라우팅 검증 가능

### 3단계 — 평가자 대시보드 (§2)
- 카드 6종 + 클릭 필터 → 테이블 컬럼 확장·기본 정렬·헤더 정렬·페이징 → 행 클릭 `/r2/review?member={id}` 라우팅(D1) → 세션 사용자 헤더
- **검증**: 카드 필터 ↔ 테이블 연동, 행 클릭 → 검토 화면에 해당 팀원 선택 상태로 진입

### 4단계 — 코칭 캘린더 (§4, 완전 독립)
- `eval_phases` 조회 연동(더미 폴백 포함) → 월 이동(◀▶)
- **검증**: 월 이동 시 일정 재조회, DB 일정 1건 추가 후 화면 반영 확인
